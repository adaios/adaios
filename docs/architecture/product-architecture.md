# AdaiOS 产品架构

Version: 1.0

---

## 五层产品架构

AdaiOS 的用户体验从下到上分为五层，每一层回答不同的用户需求：

```
┌──────────────────────────────────────────────────────┐
│  Layer 2: 主动推送                                     │
│  "你今天需要知道什么"                                   │
├──────────────────────────────────────────────────────┤
│  Layer 1: AI 问答                                     │
│  "我问，AI 答"                                         │
├──────────────────────────────────────────────────────┤
│  Layer 4: 通用记录入口                                  │
│  "不只是问答，更是记录各种事情的入口"                     │
├──────────────────────────────────────────────────────┤
│  Layer 3: 数字身份 + 记忆持久化                         │
│  "跨会话、换模型，AI 快速了解我"                         │
├──────────────────────────────────────────────────────┤
│  Layer 5: 外部信息接入                                  │
│  "我关心的世界正在发生什么"                              │
├──────────────────────────────────────────────────────┤
│  ┌────────────────────────────────────────────────┐  │
│  │  Layer 6: 交易系统反哺（垂直领域闭环）            │  │
│  │  交易日志 → 复盘沉淀 → 反哺交易知识库            │  │
│  └────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

---

### Layer 1: AI 问答

**用户感受：** "像 ChatGPT 一样直接问，但 AI 了解我"

| 能力 | 说明 |
|------|------|
| 问答 | 自然语言提问，AI 基于你的上下文回答 |
| 意图识别 | 后台自动判断是"提问"还是"记录"，不需要用户选择 |
| 今日会话 | 同一天内的对话自动关联，不需要建会话房间 |

**数据流：**

```
用户输入 → IntentRecognizer(识别意图)
             ↓
           QUESTION → Context Engine(含今日会话历史) → AI 回答
```

---

### Layer 2: 主动推送

**用户感受：** "你不问，App 也告诉你今天需要知道的"

| 能力 | 说明 |
|------|------|
| 今日简报 | 每天第一次打开时的自然语言问候 |
| 兴趣推送 | 基于你关注的领域、股票、话题推送相关内容 |
| 日程提醒 | 结合时间规划、行程的智能提醒 |
| 持仓异动 | 你关注的资产有异常波动时主动通知 |

**原则：** 推送不是骚扰，是关心。每条推送必须贴合你的兴趣和当前状态。

---

### Layer 3: 数字身份 + 记忆持久化

**用户感受：** "换了模型、新开会话，AI 依然快速了解我"

| 组件 | 职责 | 存储 |
|------|------|------|
| Identity | 你是谁（偏好、规则、标签） | `data/identity/profile.md` |
| Memory | 你的历史（AI 理解的沉淀） | `data/memory/YYYY/MM.md` |
| Knowledge | 你的知识体系（逐步构建） | `data/knowledge/`（预留） |

**核心能力：** 跨会话、跨模型的身份恢复。新模型接入时，通过 Context Engine 读取 Identity + Memory + Knowledge，快速还原对你的理解。

---

### Layer 4: 通用记录入口

**用户感受：** "不只是问答，也是各种事情的入口，不需要每次回复"

| 能力 | 说明 | 当前状态 |
|------|------|---------|
| 文字记录 | 纯文本输入，AI 判断是否需要回复 | ✅ 已实现 |
| 沉默记录 | 只需要记下来，不需要 AI 回复 | ⚠️ 部分（LOG 意图） |
| 图片/视频/音频 | 多模态记录入口 | ❌ 未实现 |

**意图判定：**

| 输入特征 | 意图 | 行为 |
|---------|------|------|
| 有疑问词、信息查询 | QUESTION | 回答，不存记录 |
| 交易决策疑问 | DECISION | 存记录 + AI 分析 + 建议 |
| 纯描述、无问句 | LOG（含 silent） | 存记录 + AI 理解（可静默） |

---

### Layer 5: 外部信息接入

**用户感受：** "我关心的世界正在发生什么"

| 能力 | 说明 |
|------|------|
| 实时热点 | 基于你兴趣领域的实时动态 |
| 新闻 | 贴合持仓、关注行业的新闻推送 |
| 市场消息 | 大盘、板块、持仓个股的异动信息 |

**原则：** 不相关的信息不推。所有外部信息经过兴趣模型和持仓筛选后再呈现。

---

## 垂直闭环：Layer 6 — 交易系统反哺

贯穿 Layer 1-5 的垂直领域闭环，连接日常交易操作和交易知识体系。

```
Layer 4 记录交易
    ↓
Layer 1 AI 理解
    ↓
data/trading/reviews/（复盘自动沉淀）   ← 自动
    ↓
你审核 → os/trading-os/（知识入库）     ← 手动
    ↓
Layer 3 Identity + Memory + Knowledge 更新
    ↓
下次 Layer 1 回答时，AI 更懂你的交易风格
```

### 数据流向

| 数据 | 来源 | 目标 | 方式 |
|------|------|------|------|
| 交易日志 | App 输入 | `data/records/` | 自动 |
| 持仓更新 | App 交易 | `data/trading/positions.md` | 自动 |
| AI 理解 | DeepSeek | `data/memory/` | 自动 |
| 复盘沉淀 | AI 分析 | `data/trading/reviews/` | 自动 |
| 知识入库 | 你审核 | `os/trading-os/` | 手动 |

## 系统分层与技术映射

```
用户层 ──────── apps/adai-app/（Flutter）

入口层 ──────── interfaces/（REST API）
                     ↓ IntentRecognizer
处理层 ──────── application/（用例编排）
                     ↓ Context Engine
核心层 ──────── kernel/（Identity / Record / Context / Memory / Knowledge）
领域层 ──────── domain/（Trading OS / Life OS / Project OS）
基础设施层 ──── infrastructure/（FileStorage / AI LLM / DB）
                  ↑
外部 ───────── os/trading-os/（知识资产，File First）
```

## 设计原则

1. **入口统一，后台分流** — App 只有一个输入口，后台自动分流到不同处理路径
2. **沉默记录** — 大量输入不需要 AI 回复，只需要记下来。减少不必要的 AI 调用
3. **身份可迁移** — 换模型、新会话，AI 通过 Identity + Memory + Knowledge 快速恢复对你的了解
4. **推送即关心** — 推送的信息必须基于你的兴趣、持仓、日程，不相关的不推
5. **知识反哺** — 日常操作沉淀的数据，经过你筛选后反哺到知识系统，让 AI 越来越懂你

## os/ 目录独立性

`os/` 下的每个项目（如 `os/trading-os/`）有独立的工作流：

- 有自己的 `CLAUDE.md`、独立的工作流程和目录规则
- AdaiOS mono repo 只是存放位置，不干涉内部运作
- 不依赖 adai-core 任何代码
- adai-core 通过文件系统单向读取其产出的知识资产（`11-context/`）
- 从不反向写入：`os/` → `data/`，不反向
- **Git 统一在根仓库管理，工作焦点各自独立**
