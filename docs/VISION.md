# AdaiOS 项目愿景与上下文

Version: 0.3

---

# 1. 项目名称

AdaiOS (Adai Personal AI Operating System)

中文：阿呆个人 AI 操作系统

---

# 2. 项目定位

AdaiOS 不是一个普通 App。它的目标不是解决单一问题，而是探索：

> 如何构建一个长期陪伴个人成长的 AI 原生数字操作系统。

传统软件围绕功能、页面、数据库、用户操作进行设计。

AdaiOS 围绕人、记忆、上下文、知识、决策、AI 协作构建个人数字基础设施。

---

# 3. 核心理念

## 3.1 人不是数据，而是持续变化的系统

传统应用：用户 → 输入 → 数据 → 功能

AdaiOS：

```
人生事件 → Record → Context → Memory → Knowledge → 理解 → 辅助决策
```

核心不是保存数据，而是让 AI 逐渐理解一个人的世界。

## 3.2 Context 是未来 AI 系统的核心

未来 AI 的竞争不只是模型能力。模型会越来越接近。真正稀缺的是 AI 是否理解你的长期上下文——目标、习惯、知识体系、历史选择、偏好、思考方式。

AdaiOS 的核心能力：建立 Personal Context Layer。

## 3.3 文件代表一切

AdaiOS 延续 Unix/Linux 哲学：Everything is a file。

个人知识不应该被封闭在某个平台。重要资产应该可读取、可迁移、可版本管理、可被 AI 理解。

Markdown 文件 + 目录结构是个人知识资产的重要载体。数据库负责运行，文件负责沉淀。

## 3.4 AI 不是工具，而是协作者

传统：人使用 AI。AdaiOS：人 + AI 长期协作。

AI 不只是回答问题，而应该理解背景、记住历史、提供建议、发现模式、辅助决策。最终形成个人数字伙伴。

## 3.5 File First, Database Second, Context Always

这是 AdaiOS 最高设计原则。

### File First（文件优先）

所有长期知识、交易系统、生活记录、项目文档，都以目录 + 文件的形式存在。Git 可管理，AI 可直接读取，人可以脱离任何软件查看。格式：Markdown + 结构化目录。

### Database Second（数据库第二）

数据库不是最终存储。数据库是为了 App 的查询速度、搜索、统计、同步和缓存而存在。数据库可以随时根据文件重新构建。**文件是 Source of Truth，数据库是 Derived。**

### Context Always（上下文永远）

任何模块（Trading、Life、Project）都不直接暴露数据库。所有模块通过 **Context Engine** 输出统一的上下文能力。这是 ACP（AdaiOS Context Protocol）真正应该解决的问题。

# 4. 五层产品架构

AdaiOS 的用户体验从下到上分为五层，每一层回答不同的用户需求：

```
┌──────────────────────────────────────────┐
│  Layer 2: 主动推送                        │
│  "你今天需要知道什么"                      │
├──────────────────────────────────────────┤
│  Layer 1: AI 问答                        │
│  "我问，AI 答"                            │
├──────────────────────────────────────────┤
│  Layer 4: 通用记录入口                     │
│  "不只是问答，更是记录各种事情的入口"        │
├──────────────────────────────────────────┤
│  Layer 3: 数字身份 + 记忆持久化            │
│  "跨会话、换模型，AI 快速了解我"            │
├──────────────────────────────────────────┤
│  Layer 5: 外部信息接入                     │
│  "我关心的世界正在发生什么"                 │
├──────────────────────────────────────────┤
│  Layer 6: 交易系统反哺（垂直领域闭环）      │
│  交易日志 → 复盘沉淀 → 反哺交易知识库      │
└──────────────────────────────────────────┘
```

## Layer 1: AI 问答

用户感受："像 ChatGPT 一样直接问，但 AI 了解我"

- 意图识别：后台自动判断是"提问"还是"记录"，用户不用选择
- 今日会话：同一天内的对话自动关联，不需要建会话房间
- 沉默记录：大量输入只需要记下来，不需要 AI 回复

## Layer 2: 主动推送

用户感受："你不问，App 也告诉你今天需要知道的"

- 今日简报：每天第一次打开时的自然语言问候
- 兴趣推送：基于你关注的领域和话题推送相关内容
- 持仓异动：关注的资产有异常波动时主动通知

原则：推送不是骚扰，是关心。每条推送必须贴合你的兴趣和当前状态。

## Layer 3: 数字身份 + 记忆持久化

用户感受："换了模型、新开会话，AI 依然快速了解我"

- Identity：你是谁（偏好、规则、标签）
- Memory：你的历史（AI 理解的沉淀）
- Knowledge：你的知识体系（逐步构建）

跨会话、跨模型的身份恢复能力是新模型接入时的核心保障。

## Layer 4: 通用记录入口

用户感受："不只是问答，也是各种事情的入口，不需要每次回复"

- 文字记录：纯文本输入，AI 自动判断是否需要回复
- 多模态：图片、音频、视频记录入口（预留）
- 意图判定：后台根据输入内容自动路由到 log / question / decision

## Layer 5: 外部信息接入

用户感受："我关心的世界正在发生什么"

- 实时热点：基于你兴趣领域的实时动态
- 新闻：贴合持仓、关注行业的新闻推送
- 市场消息：大盘、板块、持仓个股的异动信息

原则：不相关的信息不推。所有外部信息经过兴趣模型筛选。

## Layer 6: 交易系统反哺（垂直闭环）

贯穿所有层的垂直领域闭环：

```
交易记录 → AI 理解 → 复盘自动沉淀 → 你审核 → 反哺交易知识库
                                                      ↓
                                           下次 AI 更懂你的交易风格
```

| 数据 | 来源 | 目标 | 方式 |
|------|------|------|------|
| 交易日志 | App 输入 | data/records/ | 自动 |
| 持仓更新 | App 交易 | data/trading/positions.md | 自动 |
| AI 理解 | LLM | data/memory/ | 自动 |
| 复盘沉淀 | AI 分析 | data/trading/reviews/ | 自动 |
| 知识入库 | 你审核 | os/trading-os/ | 手动 |

# 5. 系统架构

## 5.1 架构总览

```
Human
  │ 输入 (App / Voice / Web)
  ▼
Record / File First
  │
  ▼
Kernel
  ├── Identity      个人档案
  ├── Record        最小事件单元
  ├── Timeline      Record 时间序列投影
  ├── Context       上下文引擎（★ 核心）
  ├── Memory        长期记忆
  └── Knowledge     结构化知识
  │
  ▼
Domain OS
  ├── Trading OS    金融交易（含研究）
  ├── Life OS       个人生活
  └── Project OS    项目管理
  │
  ▼
AI Model（Infrastructure 层）
```

## 5.2 Kernel（核心层）

| 组件 | 职责 |
|------|------|
| Identity | 个人长期稳定信息层，描述"这个人是谁" |
| Record | 最小个人事件单元，一切上层能力的事实基础 |
| Timeline | Record 的时间序列投影，自动生成 |
| Context | ★ 上下文引擎，Kernel 核心能力 |
| Memory | 跨会话持久化个人记忆 |
| Knowledge | 结构化知识资产 |

## 5.3 Domain OS（领域层）

| Domain OS | 职责 |
|-----------|------|
| Trading OS | 金融交易、策略、复盘（含研究） |
| Life OS | 日记、情绪、习惯、生活记录 |
| Project OS | 想法、规划、执行 |

## 5.4 Infrastructure（基础设施层）

AI 不是独立业务层，是基础设施。

# 6. 产品设计原则

1. **入口统一，后台分流** — App 只有一个输入口，后台自动判断处理路径
2. **沉默记录** — 大量输入不需要 AI 回复，只需要记下来。减少不必要的 AI 调用
3. **身份可迁移** — 换模型、新会话，AI 通过 Identity + Memory + Knowledge 快速恢复对你的了解
4. **推送即关心** — 推送的信息必须基于你的兴趣、持仓、日程，不相关的不推
5. **知识反哺** — 日常操作沉淀的数据，经过你筛选后反哺到知识系统

# 8. 工程原则

### 不过度设计

不要提前微服务化、分布式、Agent 平台。先建立核心。

### 模块边界优先

任何新能力必须回答：属于哪个 Domain？

- 交易分析 → Trading OS
- 生活记录 → Life OS
- AI 上下文 → Kernel/Context

### 数据资产优先

代码会变化。数据和知识是长期资产。

### File First, Database Second, Context Always

最高设计原则。详见 §3.5。

---

# 9. 长期愿景

未来 AdaiOS 希望成为一个人的：第二大脑、知识系统、决策系统、工作系统。

最终 AI 不只是回答"这个问题是什么？"，而是能够理解"这个人是谁，他正在做什么，他为什么这样选择，下一步应该如何行动。"

---

# 10. 给 AI Agent 的理解要求

当参与 AdaiOS 开发时，请始终理解：AdaiOS 不是一个普通 CRUD 应用，它是一套 Personal AI Operating System。

开发任何功能时优先考虑：
1. 是否增强个人上下文？
2. 是否沉淀长期资产？
3. 是否帮助 AI 更理解用户？
4. 是否符合五层产品架构定位？
5. **是否尊重 os/ 目录的独立性？** — `os/` 下的项目是完全独立的，adai-core 只从中读取，不写入，不干涉其内部流程。

不要只实现功能。要建设系统。
